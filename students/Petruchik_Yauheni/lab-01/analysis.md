# Анализ границ ответственности: Планировщик бюджета

## 1. Транзакционные границы

| Операция | Синхронная/Асинхронная | Откат при ошибке (Rollback) | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| **Проверка лимита и сохранение расхода в БД** | Синхронная | Да. Если БД падает при записи, транзакция отменяется. | Нет (контролируется пользователем - нажмет кнопку еще раз) | Да (Генерируется уникальный `Idempotency-Key` на клиенте при нажатии "Сохранить") |
| **Отправка push-уведомления семье** | Асинхронная | Нет. Если пуш не ушел, расход из БД не удаляется. | Да. 3 попытки через очередь (RabbitMQ/Celery) с интервалом в 1 минуту. | Да. Система хранит ID отправленного уведомления, чтобы не спамить. |
| **Обновление аналитических отчетов** | Асинхронная | Нет. | Да (повторный пересчет по крону каждую ночь) | Да |

## 2. Обработка исключительных ситуаций

### Исключительная ситуация 1: Превышение лимита (Бизнес-логика)
- **Условие возникновения:** Пользователь вводит сумму, которая вместе с текущими тратами превышает заданный на месяц лимит.
- **Обнаружение:** Backend API проверяет математику: `Текущие_траты + Новая_сумма > Лимит`.
- **Реакция:** Отклонение запроса (код 400).
- **Компенсация:** Никаких записей в БД не производится. Откатывать нечего.
- **Уведомление пользователя:** Модальное окно или красный текст: "Невозможно добавить: вы превысили лимит на X рублей".

### Исключительная ситуация 2: Падение базы данных (Технический сбой)
- **Условие возникновения:** Разрыв соединения с PostgreSQL во время сохранения транзакции.
- **Обнаружение:** Перехват исключения `ConnectionTimeout` или `SqlException` на бэкенде.
- **Реакция:** Прерывание процесса обработки. Запись ошибки в логи (Sentry).
- **Компенсация:** Транзакция БД автоматически откатывается (ACID).
- **Уведомление пользователя:** Сообщение "Произошла внутренняя ошибка сервера. Пожалуйста, попробуйте позже".
