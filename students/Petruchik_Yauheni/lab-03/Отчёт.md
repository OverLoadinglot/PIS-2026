## Цель работы

Применить тактические паттерны DDD (Entities, Value Objects, Aggregates, Domain Events, Specifications, Domain Services) при реализации **доменного слоя** с инвариантами и доменной логикой для планировщика семейного бюджета.

---

## Вариант №49 — «Планировщик бюджета семьи “Про рубли”» 💵

**Питч:** Все траты видны, ссоры в семье меньше.

**Ядро домена:** Члены семьи, Категории, Операции (траты/доходы), Лимиты, Отчёты, Совместные расходы.

## Ход выполнения работы

### 1. Value Objects

Созданы иммутабельные объекты-значения (файл `value_objects.py`):

- `Money` – сумма с валютой, поддержка сложения/вычитания, валидация неотрицательности.
- `CategoryName`, `MemberName` – строковые VO с проверкой длины.
- `DateRange` – период для лимитов и отчетов.
- `Percentage` – для возможного распределения совместных трат.
- `TransactionType` – перечисление (доход/расход).

### 2. Entities

Реализованы сущности (`entities.py`):

- `FamilyMember` – идентификация по `id`, имя как VO.
- `Category` – идентификация по `id`, название как VO.
- `Transaction` – идентификация по `id`, содержит сумму, тип, категорию, члена семьи, дату.

Все сущности переопределяют `__eq__` и `__hash__` на основе `id`.

### 3. Aggregate Root

**Корень агрегата:** `FamilyBudget`. Инкапсулирует:

- списки членов, категорий, транзакций, лимитов;
- методы изменения: `add_member`, `add_category`, `set_limit`, `add_transaction`, `archive`.

**Инварианты:**

| № | Инвариант | Где проверяется |
|---|-----------|----------------|
| 1 | Нельзя изменять архивный бюджет | все методы изменения проверяют `_is_archived` |
| 2 | Имя категории уникально в рамках бюджета | `add_category` |
| 3 | Член семьи и категория должны существовать при добавлении транзакции | `add_transaction` |
| 4 | Расход не может превысить установленный лимит категории за период | `add_transaction` (генерирует событие и выбрасывает исключение) |
| 5 | Лимит перезаписывается при пересечении периодов | `set_limit` |

### 4. Domain Events

Файл `events.py`:

- `TransactionAddedEvent` – после успешного добавления операции.
- `LimitExceededEvent` – при попытке превысить лимит (событие записывается в агрегат, затем выбрасывается исключение).
- `BudgetArchivedEvent` – при архивации бюджета.

События накапливаются в агрегате и извлекаются через `pull_events()`.

### 5. Domain Services

Реализован сервис `BudgetReportingService` (`reporting_service.py`) для построения отчётов по доходам/расходам за период с группировкой по категориям и членам семьи. Сервис не имеет состояния и оперирует только публичным API агрегата.

### 6. Specification

В `specifications.py` реализованы:

- Базовые классы `Specification`, `AndSpecification`, `OrSpecification`, `NotSpecification`.
- Конкретные спецификации: по категории, по члену семьи, по диапазону дат, по типу транзакции.
- Продемонстрировано комбинирование спецификаций (пример в тестах).

### 7. Интерфейсы портов (бонус)

- `FamilyBudgetRepository`, `TransactionRepository` – исходящие порты для работы с хранилищем.
- `EventBus` – порт для публикации доменных событий.
- `InMemoryEventBus` – адаптер для тестирования (in‑memory).

### 8. Юнит-тесты

Файл `tests/test_domain.py` содержит тесты (`unittest`):

- Добавление транзакции и генерация события.
- Проверка лимита (исключение `LimitExceededException`).
- Ошибка при несуществующем члене семьи.
- Запрет изменений в архивном бюджете.
- Работа отчётного сервиса.
- Использование спецификаций для фильтрации.

Все тесты проходят успешно.

## Контрольные вопросы

1. **VO vs Entity:** Entity имеет постоянную идентичность (id) и может меняться; VO определяется полностью своими атрибутами, неизменяем и не имеет id.
2. **Зачем инкапсуляция агрегата:** Агрегат обеспечивает согласованность изменений внутри своих границ, защищает инварианты и является транзакционной границей.
3. **Роль Domain Events:** Фиксируют важные для бизнеса факты, произошедшие в домене, позволяя другим компонентам (например, проекциям или сагам) реагировать асинхронно без связывания.
4. **Проверка инвариантов:** В конструкторах VO, в методах агрегата перед изменением состояния, в сеттерах/конструкторах сущностей.
5. **Иммутабельность VO:** Упрощает понимание кода, безопасна при использовании в коллекциях, исключает побочные эффекты.
