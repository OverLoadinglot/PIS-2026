## Цель работы

Реализовать прикладной слой с разделением на команды и запросы для семейного бюджетного планировщика.

---

## Вариант №49 — «Планировщик бюджета семьи “Про рубли”» 💵

**Питч:** Все траты видны, ссоры в семье меньше.

**Ядро домена:** Члены семьи, Категории, Операции, Лимиты, Отчёты, Совместные расходы.

---

## Ход выполнения работы

### 1. Команды (Commands)

Созданы команды в `src/application/command/commands.py`:

- `CreateBudgetCommand`
- `AddMemberCommand`
- `AddCategoryCommand`
- `SetLimitCommand`
- `AddTransactionCommand`

Валидация примитивов выполняется в `__post_init__`.

### 2. Command Handlers

Реализованы обработчики команд:

- `CreateBudgetHandler`
- `AddMemberHandler`
- `AddCategoryHandler`
- `SetLimitHandler`
- `AddTransactionHandler`

Handlers загружают агрегат из репозитория, вызывают методы домена, сохраняют изменения и публикуют события.

### 3. Queries и Read DTO

Добавлен запрос `GetBudgetByIdQuery` и DTO `BudgetReadDto` для плоского представления бюджета.

### 4. Query Handlers

Реализован `GetBudgetByIdHandler` для чтения состояния бюджета без изменения данных.

### 5. Application Service

Фасад `BudgetApplicationService` в `src/application/service/budget_app_service.py` объединяет команды и запросы.

### 6. Тестирование

Юнит-тест `tests/test_add_transaction_handler.py` проверяет:

- получение бюджета из репозитория;
- сохранение после добавления транзакции;
- публикацию событий через `EventPublisher`.

## Контрольные вопросы

1. Команда изменяет состояние, запрос только читает данные.
2. Handler команды сохраняет агрегат и публикует события, но не возвращает доменный объект.
3. Валидация примитивов выполняется в DTO команд, инварианты — в домене.
4. Для отчётов используется Read DTO, не доменные сущности.
5. Application Service объединяет логику передачи команд и запросов между портами.
